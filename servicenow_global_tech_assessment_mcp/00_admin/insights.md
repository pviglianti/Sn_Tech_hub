## Active Decisions
- **Web App = full control plane** (management, visibility, data browsing, MCP setup wizard). **MCP = reasoning plane** (AI judgment: feature grouping, tech debt analysis, disposition recommendations, CSDM service structure). AI reads local DB, uses Snow-flow for supplemental SN data, writes findings back to app tables. See `mcp_app_blueprint.md` for full architecture.
- ChatGPT/deep research compatibility is now exposed through the existing Python MCP server as standard read-only `search(query)` and `fetch(id)` tools over Tech Assessment Hub data (findings, features, recommendations, assessment reports); the JSON-RPC layer passes raw MCP `content` arrays through unchanged for these tools while preserving JSON-wrapped behavior for the legacy tool set.
- AI stage execution decision (2026-03-28): `ai_analysis` should use a connected MCP-capable model session whenever provider/auth config is available, not just cache prompt/context JSON. The connected model writes scope flags, observations, and structured `ai_observations` metadata (scope decision, rationale, related customized result IDs) through `update_scan_result`; deterministic later stages must preserve those writes and treat `is_out_of_scope=true` as a hard exclusion from observations/grouping/refinement/report counts.
- AI analysis hardening follow-up (2026-03-31): connected `ai_analysis` must be artifact-first, not assessment-batch-first. Each dispatch should receive the assessment target application/tables as the scope anchor, decide `in_scope` vs `adjacent` vs `out_of_scope` before any grouping work, write concrete functional observations plus direct related-result IDs, and only use engine signals or ServiceNow web/product lookup as supporting context. When runtime mode is enabled, `observations` must not run unless full scope triage was actually persisted for the customized result set.
- AI analysis MCP tool operational note (2026-04-04): current connected runs can hit two wrapper defects during per-artifact scope triage: `get_result_detail` may fail on some records because the tool still reads `VersionHistory.source` instead of `current_version_source`, and `update_scan_result` can return an MCP `Unexpected response type` wrapper error even when the DB write succeeds. Until fixed, operators should verify the underlying `scan_result` row before retrying or declaring the batch failed.
- AI loop rerun decision (2026-04-04): pre-report refinement should be a first-class rerun path, not just `complete -> ai_analysis`. `advance-pipeline` now allows `rerun=true` back to `ai_analysis` from any AI-loop stage (`ai_analysis`, `observations`, `review`, `grouping`, `ai_refinement`, `recommendations`, `report`, `complete`), archives prior `ScanResult.ai_observations` into top-level `pass_history`, resets completed AI phase checkpoints to `pending`, and leaves human edits/review state intact. Writers that update `ai_observations` must preserve `pass_history` and assign/retain `ai_loop_iteration` so later passes can refine from prior context instead of overwriting it.
- Assessment out-of-scope related-list decision (2026-04-04): assessment detail now treats out-of-scope records as a first-class related list concern. - Swarm execution decision (2026-04-04): `ai.runtime.execution_strategy=swarm` is now an active opt-in behavior for the connected Codex/Claude runtime path, not just a reserved label. In swarm mode the server still launches one parent CLI session per phase/chunk, but the parent prompt/CLI config explicitly tells the provider to use native subagents/agent-team workers. `ai_analysis.batch_size` is again a real execution lever and controls parent chunk size; in swarm mode it is raised to at least `ai.runtime.max_concurrent_sessions` so a parent can fan out meaningfully. ### ARCHITECTURE: Dynamic Registry is Canonical for ALL SN Table Mirroring
- **`SnTableRegistry` + `SnFieldMapping`** (in `models_sn.py`) is the ONE system for mirroring any ServiceNow table — CSDM, preflight, custom, or future modules. Renamed from `Csdm*` prefix on 2026-02-15.
- **All new SN tables MUST use this system.** Never create new hardcoded SQLModel classes for SN mirror data. Use `ensure_schema_exists()` → dictionary pull → dynamic DDL → `upsert_batch()`.
- **Physical mirror tables are shared across instances** (one `sn_sys_user` table for all instances). Data is partitioned by `_instance_id`. Columns are the union of all instances' fields.
- **Registry + field mappings are instance-specific.** Each instance has its own `CsdmTableRegistry` row and `CsdmFieldMapping` set per table. Display/query always scopes to the instance's mappings.
- **Schema drift between instances is handled** by `ensure_schema_exists()` calling `alter_mirror_table()` to ADD missing columns before every ingestion (even on early-return path).
- **Old static preflight models** (`update_set`, `instance_plugin`, `plugin_view`, `metadata_customization` in `models.py`) are legacy. Phase 4 migrates them to the dynamic system. Do not extend them.
- **The `source` column** distinguishes table origin: `"csdm"` | `"preflight"` | `"custom"`. UI groups by this.
- **Key files**: `src/services/csdm_ingestion.py` (registry creation, ingestion), `src/services/csdm_ddl.py` (DDL engine), `src/models_sn.py` (registry/mapping models), `src/web/routes/dynamic_browser.py` (browse API/UI).

### ARCHITECTURE: Phase 7 — 10-Stage Pipeline with AI Handlers + Re-run
- **PipelineStage extended to 10**: scans → engines → ai_analysis → observations → review → grouping → ai_refinement → recommendations → report → complete. 3 new AI stages added; engines run before ai_analysis so DFS has relationship data.
- **Human-Edit-as-Context principle**: AI checks for existing human content at every stage and never overwrites — only refines. AI may rewrite human content for better flow/grammar/spelling while preserving the human's core point.
- **Local-first contextual enrichment**: `src/services/contextual_lookup.py` — 6 functions: `detect_references`, `check_local_table_data`, `lookup_reference_local`, `lookup_reference_remote`, `resolve_references`, `gather_artifact_context`. Checks Fact cache → local DB (TableDefinition, ScanResult, UpdateSet) → SN via sn_client if allowed.
- **Enrichment modes controlled by property**: `observations.context_enrichment` — "auto" (default, local first + remote fallback), "always" (always remote), "never" (local only).
- **Reference detection**: Regex patterns for INC, CHG, RITM, REQ, PRB, TASK, WO, WOTASK/WOT, KB with longest-prefix-first alternation.

### PROCESS: Multi-Agent Orchestration Lessons (2026-03-05)
- **Zero-overlap file ownership enables clean parallel dev**. PM-verified file ownership map prevented merge conflicts across 3 worktrees.
- **Round-robin cross-testing is effective**: Dev-2→Task 1, Dev-1→Task 3, Dev-3→Task 2. Each dev verified another's work with source-level evidence.
- **Architect/PM feedback is a post-commit loop, not a merge gate**. Checkpoint 4 (cross-test complete) is the merge gate. Architect reads reviewer findings → posts lessons-learned → stores in memory for next sprint planning. PM updates checkpoints + backlog.
- **Dev-3 replicated Dev-1 identically** when building on shared files in its own worktree. The overlapping files were byte-identical, enabling simple copy merge strategy (dev_3 superset wins).
- **Worktree strategy**: Each dev gets isolated worktree branched from same base commit. Changes are uncommitted working directory modifications. Merge = file copy to feature branch → run full suite → commit.
